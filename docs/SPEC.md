## 1. Overview

termtyper is a browser-based typing speed test where the input material is real, unedited source code extracted from widely-used open-source repositories. Unlike conventional typing tests built on pangrams and lorem ipsum, termtyper forces users to contend with actual developer muscle-memory: symbols, indentation, `snake_case`, `camelCase`, generics, and the existential dread of a 40-character function signature.

> **Core Premise:** Your WPM on English prose is irrelevant. Your WPM on code is your actual bottleneck. termtyper measures the thing that matters.

---

## 2. Goals & Non-Goals

### 2.1 Goals

- Deliver an accurate, low-latency typing test experience entirely in the browser.
- Source code snippets from real, recognizable OSS repositories to give the pain meaning.
- Support five languages at launch: Python, JavaScript/TypeScript, Go, Rust, C/C++.
- Serve both developers improving raw speed and learners building syntax familiarity.
- Provide immediate, per-session feedback: WPM, accuracy, error heatmap, character-level breakdown.
- Require zero sign-up. No accounts, no tracking, no cookies beyond ephemeral session state.
- Be fast to load (sub-2s LCP on a decent connection) and keyboard-navigable end-to-end.

### 2.2 Non-Goals (v1)

- User accounts, profiles, or persistent leaderboards.
- Multiplayer or real-time competitive modes.
- Mobile / touch input optimization (keyboard-first product).
- AI-generated or synthetic code snippets.
- Snippet submission by users.

## 3. Target Audience

| Persona | Description |
|---|---|
| **Speed-focused dev** | Experienced engineer who already types well on prose but suspects their code typing is the bottleneck. Wants hard data and wants to improve it. |
| **Syntax learner** | Developer learning a new language (e.g., Go or Rust) and wants passive exposure to idioms, patterns, and naming conventions via repetition. |
| **Competitive typist** | Power user who will come back daily, track personal bests, and want difficulty tiers and filter controls. |

## 4. Code Snippet Sourcing

### 4.1 Source Repositories

Snippets are pre-curated and bundled at build time — no runtime calls to GitHub. This eliminates latency, avoids rate-limiting, and guarantees content stability.

| Language | Primary Source Repos |
|---|---|
| **Python** | CPython stdlib, Django, Flask, requests, numpy |
| **JavaScript / TypeScript** | React, Node.js core, TypeScript compiler, Prettier, ESLint |
| **Go** | kubernetes, go stdlib, gin, cobra, Hugo |
| **Rust** | tokio, ripgrep, rustc, serde, axum |
| **C / C++** | Linux kernel, SQLite, CPython C-ext, LLVM, Redis |

### 4.2 Snippet Extraction Rules

A custom offline pipeline (Node.js script, run at build time) walks repo source trees and extracts candidate snippets. Snippets **must** satisfy all of:

- **Length:** 80–300 characters (typing test sweet spot — enough challenge, short enough to finish).
- No blank-only lines within the snippet (consecutive whitespace becomes ambiguous in the test UI).
- No auto-generated code (detected by header comments: `DO NOT EDIT`, `generated by`, etc.).
- No license header blocks.
- UTF-8 clean — no non-printable characters.
- **License:** Permissive only — MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, ISC. **GPL, LGPL, AGPL, and EUPL are explicitly excluded.** Copyleft licenses impose obligations on derivative works incompatible with a commercial web product. The pipeline reads each repo's SPDX identifier and skips any file in a GPL-family repo entirely — not just the license header.

> **License Policy:** When in doubt, skip it. The corpus has 2,000 slots. There is no shortage of MIT/Apache-licensed code to fill them. GPL snippets are not worth the legal exposure.

### 4.3 Snippet Curation Tiers

Each snippet is tagged with a difficulty tier, computed by the pipeline. Users select their preferred tier upfront on the Config screen (see §6.1).

| Tier | Heuristic |
|---|---|
| **Easy** | Mostly alphanumeric, low symbol density (<15%), no generics or macros. |
| **Medium** | Moderate symbol density (15–35%), some nesting, standard operators. |
| **Hard** | High symbol density (>35%), generics, macros, regex literals, or deeply nested expressions. |

> **Design Note:** Difficulty is a first-class user choice, not a gate. Users choose their own suffering level before the test begins.

## 5. Typing Engine

### 5.1 Input Model

- The test renders a fixed code snippet character-by-character in a monospace font.
- The user types into a hidden input field; keystrokes are captured via keyboard events.
- Characters are visually highlighted as: pending (default), correct (green), incorrect (red), current cursor position (blinking underline or block cursor).
- Backspace is supported and corrects the previous character (error is still counted for accuracy).
- **Tab is a single character.** One Tab keypress = one character in the WPM calculation, regardless of how many spaces it visually represents. This is consistent with how developers actually type (one key = one action) and prevents WPM deflation on indentation-heavy languages like Python and Go. The display renders the tab as its natural visual indent width; the engine counts it as 1.
- The test begins on first keypress (no start button).
- The test ends when the last character of the snippet is typed correctly.

### 5.2 Whitespace Handling

Code whitespace is the hardest part to get right. Rules:

- Leading spaces/tabs on each line are displayed as visible guide dots (`·`) or ghost characters so the user knows exactly how many spaces to type.
- Newlines are represented as a visible `↵` character; the user presses Enter to advance.
- Trailing whitespace on lines is stripped at extraction time — it's a trap, not a challenge.

### 5.3 Metrics Computed

| Metric | Definition |
|---|---|
| **WPM (Gross)** | Total characters typed / 5, divided by elapsed minutes. Standard definition, no error penalty. |
| **WPM (Net)** | Gross WPM minus (errors / elapsed minutes). The number that matters. |
| **Accuracy %** | (Correct keystrokes / total keystrokes) × 100. |
| **Error Map** | Per-character error frequency across the snippet, visualized as a heat overlay. |
| **Consistency** | Coefficient of variation of inter-keystroke intervals. High consistency = smooth, low-variance rhythm. |
| **Time to First Error** | Elapsed time before the first mistake. Proxy for how long the user can maintain focus. |

### 5.4 Timer Behavior

- Timer starts on first keypress, stops on snippet completion.
- No countdown timer in v1 — the test is snippet-length-bounded, not time-bounded.
- Elapsed time is displayed in real-time (`MM:SS.ms` format).

## 6. UX & Interface Design

### 6.1 Page Structure

Single-page application. No routing needed in v1. Four states:

- **Config** — User selects language and difficulty before the test begins. This is the landing state. No snippet is shown yet. Selection is required, not optional.
- **Idle** — Snippet displayed based on chosen language/difficulty, waiting for first keypress. Repo and length sub-filters are available here as optional refinements.
- **Active** — Timer running, character highlights updating, live WPM shown (updated every 500ms).
- **Results** — Full metrics displayed; options to retry same snippet or get a new one (config preserved).

> **UX Principle:** Language and difficulty are first-class choices, not sidebar filters. The user makes them once on landing, then the product gets out of the way. Changing them mid-session requires explicitly returning to Config state.

### 6.2 Layout

- Dark mode by default (it's a code tool — light mode is the opt-in deviation).
- **Config screen:** Large, tappable language cards (Python, JS/TS, Go, Rust, C/C++) and a difficulty row (Easy / Medium / Hard / Any). Single-select each. A 'Start' button (or pressing any key) loads a snippet and enters Idle. No other chrome on this screen.
- Full-width snippet display area in a monospace font (JetBrains Mono via Fontsource, self-hosted — no FOUT).
- Font size: 16–18px with 1.6 line height. No horizontal scroll — snippet wraps at word boundaries.
- In Idle state, optional sub-filters (Repo, Length) are available as compact pill toggles above the snippet. Language and difficulty are shown as read-only badges — click them to return to Config and change.
- Live stats bar (WPM, accuracy, timer) is pinned below the snippet during Active state.
- Results panel slides up from the bottom on completion (no page reload).
- **Snippet display scrolls automatically** as the user types — the current character line is always vertically centered in the display window (IDE cursor behavior). The user never manually scrolls. Lines above the cursor fade to 40% opacity to preserve spatial context without distraction.
- Maximum visible lines in the display window: 6. Snippets taller than this scroll; snippets shorter are top-aligned.

### 6.3 Keyboard Navigation

| Key | Action |
|---|---|
| `Esc` (in Idle / Active) | Return to Config screen to change language or difficulty |
| `Tab` (on Config screen) | Cycle through language / difficulty options |
| `Enter` (in Results) | Load new snippet with same language and difficulty |
| `Ctrl+R` / `Cmd+R` | Blocked — custom retry logic prevents accidental reload during active test |

### 6.4 Accessibility

- ARIA live regions for screen-reader-compatible WPM updates.
- Sufficient color contrast for all text states (correct / incorrect / pending) — WCAG AA minimum.
- No motion animations during the active test; `prefers-reduced-motion` media query respected.

## 7. Results Screen

The results screen is the product's payoff. It should feel like a post-match breakdown, not a toast notification.

- **Headline metrics:** Net WPM, Gross WPM, Accuracy % — large, immediate, no hunting.
- **Character heatmap:** the full snippet re-rendered with per-character error frequency shown as a background color intensity overlay (0 errors = neutral, 1+ errors = increasing red saturation).
- **Slowest characters:** Top 5 characters the user was slowest on (by inter-keystroke delay).
- **Consistency score:** Shown as a single percentage. Above 80% = good rhythm. Below 60% = "you're mashing keys."
- **Snippet attribution:** filename, repo, and a direct link to the source on GitHub.
- **CTA buttons:** 'New Snippet' (same filters) and 'Retry This Snippet'.

## 8. Snippet Filter System

Filters are stateless and client-side. Language and difficulty are locked in at Config; Repo and Length are optional refinements available in Idle state.

| Filter | Where Set | Options |
|---|---|---|
| **Language** | Config screen (required) | Python · JavaScript · TypeScript · Go · Rust · C · C++ |
| **Difficulty** | Config screen (required) | Easy · Medium · Hard · Any |
| **Repo** | Idle state (optional) | Dropdown of repos matching selected language |
| **Length** | Idle state (optional) | Short (80–150) · Medium (151–225) · Long (226–300 chars) |

> **Edge Case:** If the active filter combination yields fewer than 5 snippets, warn the user. If it yields 0, auto-relax the Repo filter first, then Difficulty.

## 9. Technical Architecture

### 9.1 Stack

| Layer | Technology |
|---|---|
| **Frontend framework** | SvelteKit — lightweight, no VDOM overhead (important for keystroke latency) |
| **Styling** | Tailwind CSS + CSS custom properties for theme tokens |
| **Monospace font** | JetBrains Mono via Fontsource (self-hosted, no FOUT) |
| **Snippet corpus** | Static JSON, gzipped, bundled at build time via Vite asset pipeline |
| **Hosting** | Cloudflare Pages (global edge, zero cold starts, free tier covers it) |
| **Build pipeline** | Vite + custom Node.js snippet extraction scripts |
| **Analytics (optional)** | Plausible or Fathom — privacy-first, no personal data, opt-in |

### 9.2 Corpus Delivery & Update Strategy

The corpus is updatable without a full frontend redeploy.

- `corpus.json` is hosted at a stable, versioned CDN path (e.g., Cloudflare R2): `/corpus/v{N}/corpus.json`.
- **Version manifest:** `/corpus/latest.json` contains `{ "version": N, "url": "..." }`. The frontend fetches this on load (max-age: 1 hour) to discover the current corpus version.
- **Bundled fallback:** A pinned corpus version is always bundled inside the frontend build. If the manifest fetch fails (offline, CDN outage, network error), the app silently falls back to the bundled corpus. Zero broken sessions.
- **Cache strategy:** `corpus.json` URLs are immutable (`Cache-Control: max-age=31536000, immutable`). Only `latest.json` has a short TTL. Old version URLs stay valid forever.
- **Update workflow:** run the extraction pipeline → bump version → upload new `corpus.json` to CDN → update `latest.json`. No frontend redeploy. Corpus refresh takes ~5 minutes.

> **Fallback Rule:** The bundled corpus is the floor, not the ceiling. Users always get a working product. The CDN corpus is always the preferred source when reachable.

### 9.3 Snippet Corpus Build Pipeline

1. Clone/fetch target repos at pinned commits (reproducible builds).
2. Filter files by SPDX license identifier — skip any GPL-family repo entirely (see §4.2).
3. Walk remaining source files, extract candidate snippets per §4.2 rules.
4. Score each snippet for difficulty tier per §4.3 heuristics.
5. Deduplicate (Jaccard similarity threshold: >0.85 = duplicate, drop the newer).
6. Emit `corpus.json` — array of snippet objects (see Appendix A for schema).
7. Gzip and upload to CDN under `/corpus/v{N}/corpus.json`; update `/corpus/latest.json`. Also bundle into `/static` as fallback.

> **Target Corpus Size:** ~2,000 snippets at launch (~400 per language). Enough variety to avoid repetition for several sessions; small enough to bundle without pain.

### 9.4 Keystroke Latency Budget

The typing engine is the heart of the product. Every millisecond of input lag is a bug.

- Keystroke handler must complete in <1ms (no DOM thrashing on keypress).
- Character state is maintained in a `Uint8Array` / plain array — not reactive state.
- DOM updates batched via `requestAnimationFrame`, not triggered per-keypress.
- No network calls during an active test session.

## 10. Performance Requirements

| Metric | Target |
|---|---|
| **LCP** (Largest Contentful Paint) | < 2.0s on 4G connection |
| **TTI** (Time to Interactive) | < 2.5s |
| **Keystroke-to-render latency** | < 16ms (one frame at 60fps) |
| **Corpus JSON** (gzipped) | < 500KB |
| **Total page weight** | < 800KB (JS + CSS + fonts) |
| **Lighthouse Performance score** | ≥ 90 |

- Global or per-language leaderboards (requires accounts).

## 12. Success Metrics

Since there are no accounts, success is measured via anonymized aggregates (Plausible-style analytics) and qualitative feedback.

| Metric | Target |
|---|---|
| **Session completion rate** | >60% of users finish at least one snippet |
| **Retry rate** | >40% of completed sessions trigger 'Retry' or 'New Snippet' |
| **Bounce rate** | <30% leave within 10 seconds of load |
| **Median session length** | >3 minutes (suggests multiple rounds) |
| **Social shares / organic links** | Qualitative — no target, just tracked |

## Appendix A: Snippet JSON Schema

Each entry in `corpus.json` conforms to:

```json
{
  "id": "rust_tokio_0047",
  "lang": "rust",
  "repo": "tokio-rs/tokio",
  "file": "tokio/src/runtime/mod.rs",
  "difficulty": "hard",
  "length": 187,
  "chars": "pub fn block_on<F: Future>...",
  "attribution_url": "https://github.com/tokio-rs/tokio/blob/.../mod.rs#L214"
}
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier: `{lang}_{repo_slug}_{index}` |
| `lang` | string | One of: `python`, `javascript`, `typescript`, `go`, `rust`, `c`, `cpp` |
| `repo` | string | Source repo in `owner/name` format |
| `file` | string | Relative path within the repo |
| `difficulty` | string | One of: `easy`, `medium`, `hard` |
| `length` | number | Character count (Tab counts as 1) |
| `chars` | string | The raw snippet text |
| `attribution_url` | string | Direct GitHub permalink to the source line |
